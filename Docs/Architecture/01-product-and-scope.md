# 01. プロダクト定義とスコープ

## プロダクト名

**Manufacturing Triage Agent**

## 提供価値（MVPで絶対に見せる4点）

単なる「検索」ではなく、異常入力に対して以下を **1画面で** 返す：

1. **緊急度**（High / Medium / Low）
2. **原因候補トップ3**（根拠・確信度つき）
3. **最初に確認すべき項目**（順序つき）
4. **次に誰へ渡すべきか / 何をすべきか**（エスカレーション・推奨アクション）

> 検索結果一覧を上に出すと弱い。結果画面の最上部に大きく「**まず何をするか**」を出すのが設計の肝。

### 加点強化で追加する価値（入賞狙い）

4点に加え、テーマ「鮮やかに**解決**」に応えるため次を提供する（詳細 [06-scoring-strategy.md](./06-scoring-strategy.md)）：

5. **アクション実行**：緊急度 High 等で、Agent が保全チームへ **Teams 通知/エスカレーション**を自律実行（助言→解決）
6. **画像からの判断**：故障部位の**写真を解析**（GPT-4o vision）してトリアージ精度を上げる
7. **業務改革の定量化**：**ダウンタイム削減 ROI** を可視化（画面5・記事）

## 差別化ポイント

- **チャットUIにしない**。製造現場向けの「入力フォーム + 自由記述」。現場の実務感を出す。
- **フィードバックループ**がある。現場担当が「実際の原因 / 復旧時間 / AIは当たっていたか」を登録 →
  次回の検索対象に入る。＝「静的RAGではない」「使うほど賢くなる」を実証できる。

## スコープ：作らないもの（明確に切る）

- リアルタイム IoT 連携 / 実機接続
- 本格的な音声解析
- 複雑な権限管理
- 本格的な学習基盤（再学習・ファインチューニング）
- 完全自動更新ワークフロー

## スコープ：作るもの（これだけに絞る）

- 現場入力 UI
- 文書検索 + RAG
- AI によるトリアージ出力
- 類似事例表示
- 現場フィードバック入力
- フィードバックを次回活用する簡易仕組み

## 主シナリオ

現場担当がラインで異常を発見：

```
設備:   第2ライン 搬送コンベア (L2-CONV-01)
症状:   搬送部から異音 / 温度上昇あり
コード: E-142
文脈:   直前に段取り替えあり
```

→ AI が返す：

```
緊急度:   High（品質影響の可能性あり）
原因候補: 1. 搬送ローラー摩耗  2. センサー位置ズレ  3. 条件変更による負荷上昇
初動確認: 1. ローラー部摩耗確認  2. センサー固定位置確認  3. 直前変更条件の確認
類似事例: 2025/11/04 第2ライン 同様異音 → ローラー交換で25分復旧
推奨アクション: 保全へエスカレーション / 当該ロット隔離確認
```

→ 担当者がフィードバック登録：「実際の原因 = ローラー摩耗」「復旧時間 = 22分」
→ 次回以降の判断材料になる。

## 画面構成（5画面）

| # | 画面 | 役割 | UIのポイント |
|---|------|------|------------|
| 1 | 異常入力フォーム | 設備 / 工程 / エラーコード / 症状カテゴリ / 自由記述 / **画像(B: vision解析)** | チャットにしない。フォーム + 自由記述。画像は加点要素Bで実解析 |
| 2 | **トリアージ結果**（最重要） | A.緊急度 B.原因候補Top3 C.初動確認 D.類似事例 E.推奨アクション + **保全へエスカレーション実行ボタン(A)** | 最上部に「まず何をするか」を大きく。High時はTeams通知を促す/実行 |
| 3 | 根拠詳細 | 参照した手順書 / 過去トラブル / 設備仕様 / 品質記録 | ベタ出しせず該当箇所をハイライト（citation） |
| 4 | フィードバック登録 | 実原因 / 実施対処 / 復旧時間 / AI的中可否 / メモ | これがあるだけで「静的でない」が伝わる |
| 5 | ナレッジ集計ビュー | よくある異常 / 設備別傾向 / 復旧時間上位 / 最近の知見 | "未来の広がり"を見せる。MVPは簡易集計で十分 |

## 症状カテゴリ（入力の選択肢）

異音 / 停止 / 温度異常 / 品質不良 / 振動 / その他
