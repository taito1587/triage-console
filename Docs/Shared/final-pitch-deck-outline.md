## 1. 勝ち筋の仮説 (調査統合)

### 1-1. テーマからのバックキャスト
大会テーマは **「業務の"課題"をAIエージェントでいかに"鮮やかに"解決するか」=企業の業務改革**。
鮮やかさ = ①課題が現場リアル ②エージェントが本当に「解決」している ③数字でインパクトが分かる、の3点が同時に成立すること。

### 1-2. 8分ピッチの黄金構造 (受賞作・標準ガイドの統合)

| 出典 | 提言 |
|---|---|
| Circles.Life ガイド (5分ピッチ) | Intro(30s) / Problem(30s) / Impact(30s) / Product(1m) / **Demo(2m)** / Wrap(30s) ── **デモに 40%** |
| SlideTeam / Inknarrates | スライドは **6-10枚** / 1メッセージ/枚 / アイコン > 箇条書き |
| 第1回 Zenn AI Agent Hackathon 振り返り | デモ動画は **3分以内**、ライブは1.5-2分、課題設定の解像度が決定要因 |
| Microsoft AI Agents Hackathon 2025 受賞作 (Apollo / RiskWise) | **Microsoft AI 技術が "core" であって "おまけ" でない**、citations / 構造化出力で品質を担保、判断の説明可能性 |
| ハッカソン勝者通則 | **冒頭 30 秒で主張が立たないと負け** / 技術詳細は Q&A 行き / 「動いている」を見せる比重を最大化 |

**この製品への翻訳**:
- **8 分 → 9 スライド** (平均 53 秒/枚、密度が呼吸できる)
- **デモは 2 分 (25%)**、シーンは 5 つに絞る
- **Problem + Demo で 35%** (= 2:50) を確保し、ここで勝負を決める
- **技術論パートは 1 スライドに統合** (HITL/安全/反駁を 1 枚に)
- **アーキ説明は WHY を厚く** (Foundry / 4 エージェント分割 / Cosmos Serverless / SWA+App Service の選定理由)
- **冒頭で製品名を出さない** ── Hook → 課題分解 → 製品名 reveal の順で着地

### 1-3. Microsoft 系の評価傾向
Microsoft 系ハッカソンの公開講評では **Innovation / Impact / Technical Execution / Theme Alignment** の 4 軸が繰り返し言及されている。Apollo (Best Agent in C#) の評価ポイントは「**state machine + async pipeline + citations 強制**」 ── 我々の対応物は「**connected agents + 評価ハーネス + citation必須**」。同じ設計言語で語れる。

### 1-4. 我々の強みと弱み (誇張しない)
**強み**
- 「**正しい人(保全)に、正しい情報を、正しい順番で渡す**」という、製造業の実情に合った価値プロップ (オペレーターが自力で直す前提ではない)
- **4 段階の自律性** が実装済み: Critical=auto-fire / High=承認/ Medium=並行作業 / Low=自己解決 (「時間勝負 ↔ 承認」の構造的解消)
- Foundry connected agents (主 + 品質影響スペシャリスト + 保全プランナー) + 実行トレース可視化
- **保全への情報パッケージ** (推奨工具リスト・過去事例・citation) を Teams 通知に同梱
- **Trust band** で出力の groundedness を簡易判定 (red は自動で人に回す)
- 評価ハーネスで「賢くなる」を **groundedness/正答率で定量実証**
- マルチモーダル (Vision で故障部位写真)
- 本番デプロイ済み (動かないリスクなし)
- Edge / オンプレ (Azure Local・Foundry Local) への拡張展望

**弱み (本選で突かれる前提で先手を打つ)**
- コーパスが合成データ (実マニュアルではない)
- 実工場での現地検証なし
- 精度評価は小規模テストセット
- 4 段の境目 (Critical / High の判定基準) は LLM 任せで明示ルール化はまだ

> 弱みは「**今後の検証パイプラインまで設計済み**」として転化する (Foundry Evaluation の groundedness を継続評価する図を入れる)。

### 1-5. 発表者の立ち位置 (口調と導入の設計原則)
発表者は **製造業に近い案件にエンジニアとして入った経験のある作り手**。工場ライン作業者では*ない*。
「学生です」と自己紹介はしない。**ニュートラルな "私たち" 口調** で、**事実と公開資料の数字で淡々と** 話す。

**Do**
- 「**私たちが着目したのは〜**」 / 「**公開資料の数字で見ると〜**」と、**外から見た構造の話** として入る
- 主張は **公開資料 (Siemens, 経産省ものづくり白書, Aberdeen 等) で裏付ける**
- "案件で見た" は **Q&A で問われたら答える** カードとして温存。冒頭の足場としては使わない

**Don't**
- 「**学生ですが**」のような自己ラベルから入らない
- 「**深夜の現場で〜**」のような当事者シーン描写を冒頭に置かない
- 「**現場の方は分かるはずです**」のような業界インサイダー口調は禁止
- 「**1 分あたり ¥◯◯万**」のような **聞き飽きたフレーズ単発のドカン** はやらない (製造業の審査員にとって浅い)

> 進め方は記事 (ARTICLE.md) の流れに合わせる:
> **「私たちが着目した課題は X」 → 「数字で見ると Y」 → 「ボトルネックは Z」 → 「だから作った」**
> ── 演出ではなく構造で説得する。

## 3. スライド構成 (7 枚案)

### Slide 1 — 着目した課題 (0:30)
**目的**: 業務改革テーマに対し、**自分たちが解こうとした課題を淡々と名指し** する。記事 (ARTICLE.md 1-1) の流れをほぼそのまま、ただし演出は最小に。
**話す内容 (3 段)**:

**[0-10s] 課題の名指し**
- 「私たちが着目したのは、**製造業の "計画外停止"** です。」
- 「世界の Fortune Global 500 で、計画外停止による損失は **年間 約 1.4 兆ドル、売上の約 11%** ── Siemens の 2024 年調査の数字です。」

**[10-20s] 構造を 1 段降ろす**
- 「日本国内では、技能継承に "取り組んでいる" 事業所が **92.1%** ── それだけ現場は **判断の属人化** に危機感を持っています (経産省『2024 年版ものづくり白書』)。」

**[20-30s] 着地: 本当の問題はどこか**
- 「問題は、**ラインが止まること** そのものではありません。」
- 「── **止まってから、動き出すまで** が遅いことです。」
- 「**ナレッジは社内にあります。過去事例、手順書、設備仕様 ── 全部ある。でも、必要な瞬間に必要な人に届かない。**」
- 「これが解こうとした課題です。」

**ビジュアル**:
- 上段: 2 つの数字を横並びで配置
  - **1.4 兆ドル/年** (世界の計画外停止損失) / 下に小さく出典: Siemens 2024
  - **92.1%** (技能継承に取り組む事業所) / 下に小さく出典: 経産省ものづくり白書 2024
- 下段: 命題 1 行で **「ナレッジは社内にある。でも、必要な瞬間に届かない。」**

**スピーカーノート**:
- 声を作らない。**説明会のテンション**で淡々と
- 数字を口にしたあと 1 秒の "間"
- **NG**: "ベテランの方は〜" / "現場の方なら〜" (中の人を装う)
- **OK**: "私たちが着目したのは〜" / "公開資料の数字で見ると〜"

> ⚠️ 数字は ARTICLE.md 1-1 と完全に整合。Siemens / 経産省ものづくり白書から引用。**創作した "¥◯◯万/分" は使わない**。
> 製造業の審査員が聴衆にいる前提で、**聞き飽きた数字フレーズ (1分あたり◯万) を冒頭に置かない**。Siemens の年間損失と白書の 92.1% の組み合わせは "聞き飽きていない" 数字。

### Slide 2 — 3 つの価値 → 製品 (0:40) 【新本質の中心】
**目的**: **3 つの価値** で解いている宣言 → 製品名 reveal。**ピッチ全体の背骨**。

**話す内容 (2 段)**:

**[0-25s] 縮めているのは "保全の診断時間" → 3 つの価値で解く**
- 「AI が縮めているのは、**オペレーターの判断時間ではありません**。保全担当が到着してから直すまでの **"診断時間" 15-30 分** です。」
- (画面: 3 つの価値カード)
- 「私たちは **3 つの動作** で解いています。」
  1. **正しい振り分け** ── 誤エスカレを抑える
  2. **保全への情報パッケージ** (推奨工具 + 過去事例 + Top3 原因を Teams に同梱)
  3. **オペレーターの並行作業** (待ち時間の有効化)

**[25-40s] 製品名 reveal**
- 「これを実装したのが、**Manufacturing Triage Agent** です。」
- (結果画面のスクショに切り替え)
- 「**緊急度 / 原因Top3 / 初動 / 推奨工具 / 並行作業 / エスカレ先** を **1 画面で** 返します。動かします。」

**ビジュアル (2 段切り替え)**:
- [0-25s]: 流れ図 `[異常] → [保全到着] → [診断 (赤枠 = 15-30分)] → [復旧]` + **3 つの価値カード** を横並び
- [25-40s]: 結果画面のスクショ。「**保全への情報パッケージ**」を赤枠

**話法**:
- 「**オペレーターの判断時間ではありません**」 を強く言い切る ── ナラティブの主軸
- 3 つの価値はテンポよく、**1 つあたり 5-6 秒**
- 「**動かします**」で即デモへ

> 1 文 30 字で「**保全の診断時間を縮める。振り分け × 情報パッケージ × 並行作業。**」が見えれば成功。

### Slide 3 — デモ (1:30) 【最重要・ピッチの主戦場】
**目的**: 「動いている」「鮮やかである」を体感させる。**Vision 自動入力 → Shadow mode で Teams 通知の中身** が二大クライマックス。

**シナリオ**: E-142 異音 (High判定。**写真から自動入力**で開始)

| 秒 | シーン | 見せ所 | 口頭で言うこと |
|---:|---|---|---|
| 0:25 | **📷 写真から自動入力** (設備銘板 + HMI エラー画面を映した1枚をドロップ → フォームが次々埋まる) | Vision クライマックス | 「**スマホで 1 枚撮るだけ。型式銘板も HMI エラー画面も AI が読んでフォームを自動で埋めます**。入力ゼロで診断に直行」 |
| 0:20 | トリアージ結果 (緊急度High / 原因Top3 / 初動 / **推奨工具** / **並行作業**) | "1 画面で返る" + 推奨工具と並行作業 | 「**情報パッケージと並行作業、両方ここで返ります**」 |
| 0:10 | 原因クリック → **参照資料ハイライト** + **Trust band 表示** | 出典強制 + 信頼度 | 「主張ごとに **出典必須**。Trust red は自動で人に回します」 |
| 0:30 | High → **Shadow mode で送信プレビュー** → 承認 → **Teams 通知が届く** | **🔥 クライマックス** | 「**送る前にこれが届きますと見せる**。推奨工具リストと過去事例が全部入っている。保全は到着直後から動けます」 |
| 0:05 | (口頭のみ) Critical/Low の動作差 | 4 段の自律性へのブリッジ | 「Critical なら承認スキップで即発火、Low は通知しません」 |

**fallback**: 録画デモ (45 秒) を別タブに常駐。Teams 通知の画面が含まれること必須。

**スピーカーノート**:
- 操作中に「無言時間」を作らない。手を動かす前に何を見せるか宣言
- **シーン 4 で Teams 通知を映す瞬間に 2 秒の間** ── 聴衆に推奨工具リストの中身を読ませる
- シーン 5 は **操作せず口頭のみ**、即 Slide 4 へ繋ぐ

> ⚠️ シーン 1 (Vision 自動入力) と シーン 4 (Shadow mode → Teams) は絶対に省略しない。
> 詰まったらシーン 3 (citation) を 5 秒に圧縮可能。

### Slide 4 — なぜこの設計 + アーキ (0:45) 【Technical Execution】
**目的**: 「**ただの RAG じゃないの? なぜこのスタック?**」 に **設計判断 4 つで答える**。アーキ図を背景に WHY を口頭で重ねる。

**話す内容**:
- 「**ただの RAG ではなく Agent にした理由は、自律分岐と外部ツール呼び出しが必要だから。**」
- (アーキ図を出して、4 つを順に口頭で指差し)

| 選択 | なぜ (1 文ずつ) |
|---|---|
| **Foundry Agent Service** | connected agents が native + 実行トレースが標準装備 + **Agent Service 固定費ゼロ** + Foundry Local で Edge へ移植可 |
| **4 エージェント分割** | context 最小化で幻覚率↓ + 説明可能性 + **Orchestrator が緊急度で動作を動的切替** (= Slide 5 の根拠) |
| **Cosmos Serverless** | spike-y access に合う + 運用コスト ¥0 |
| **SWA + App Service** | 1 ホスト配信でデプロイ単純 + B1 で十分 |

- 「主が **自分で専門エージェントを呼び、自分で外部ツールを叩く**。**実行ステップは Foundry Observability に全部残ります**。」

**ビジュアル (1 枚のアーキ図)**:
- 左: User → SWA / FastAPI
- 中央 (最大の塊): **Foundry Agent Service** に connected agents 4 体 (Intake / Retrieval / Orchestrator / Specialists)
- 右上: AOAI (GPT-4o + embeddings + Vision)
- 右下: Cosmos Serverless / 右端: Teams Webhook
- 矢印 8 本以内、3 色凡例 (制御 / データ / アクション)

**話法**:
- 4 つはテンポよく、1 つ 6-7 秒
- 「**自分で呼び、自分で叩く**」を 1 度だけ強く ── Agent の本質

> ⚠️ アーキ図は事前確定。当日 mermaid ライブ生成は NG。
> 詳細(SKとの比較・Cosmos の代替案など)は Q&A 行き。

### Slide 5 — 4 段階の自律性 (0:50) 【時間勝負 × 承認の両立】
**目的**: 「**時間勝負と承認は矛盾では?**」 を **4 段表 1 枚で構造的に解消**。プロダクトの設計のクライマックス。

**話す内容**:
- 「**"時間勝負と言ったのに承認で待つ? 矛盾では?"** ── 答えは、**緊急度で動作を切り替えています。実装済みです。**」
- (画面: 4 段表を指して順に)

| 緊急度 | 動作 | 内部状態 |
|---|---|---|
| **Critical** (火災・煙・人身) | **承認スキップ・即時 Teams 通知** | `auto_escalated` |
| **High** (品質影響大・単一停止) | **承認 5 秒 → Teams 通知** | `awaiting_approval` |
| **Medium** (軽微・予兆) | **承認 + 並行作業を提示** | `awaiting_approval` + 並行作業 |
| **Low** (誤検知レベル) | **通知なし、自己解決ガイドのみ** | `self_help` |

- 「**Critical は即発火、Low は通知しない**。両端で対称、間は段階的。」
- 「さらに **Trust band** (red/yellow/green) で出力の groundedness を簡易判定。**red は緊急度問わず人に必ず回ります**。」
- 「── **"AI 任せ" でも "全部人任せ" でもない、緊急度で動作差をつけている。**」

**ビジュアル**:
- 中央: **4 段の自律性表** (各行に色: Critical 赤 / High 橙 / Medium 黄 / Low 緑)
- 下: **Trust band 凡例** (red/yellow/green) を 1 行
- タイトル命題形: 「**任せきらない。待ちきらない。緊急度で切り替える。**」

**話法**:
- 「**矛盾では?**」で **聴衆と目を合わせる**
- 4 段は **Critical と Low の両端を強く**
- Trust band は 1 文で済ます

> ⚠️ コードと整合 (`backend/incident.py` の `auto_escalated / awaiting_approval / self_help`)。
> 「責任の所在」「モデルが古くなる」「賢くなる」 は **質疑で深掘り対応** (§5 Q&A 参照)。

### Slide 6 — ROI + 完成度 + Edge展望 (0:30) 【Impact + 本気度 + スケール、3 つを 30 秒で】
**目的**: テーマ "業務改革" を **数字で回収** + 「本気で作ってある」を宣言 + Edge スケールの絵で締める。1 枚で 3 役。

**話す内容 (3 段)**:

**[0-15s] ROI (3 つの価値を金額に)**
- 「インパクトは、Slide 2 の **3 つの価値ごとに控えめ試算**。」
  1. **保全診断時間 15-30 分短縮** (Aberdeen 2.5 万ドル/時換算)
  2. **誤エスカレ 約 ◯% 削減**
  3. **並行作業で復旧時間 約 ◯ 分短縮**
- 「合計、1 工場で **年間 約 ¥◯◯**。」

**[15-25s] 今出来てる**
- 「**本番デプロイ済み、実コスト数百円で稼働中**。**4 段自律性 + Trust band + 監査ログ も全部実装済み**です。」

**[25-30s] ここから伸びる**
- 「**Azure Local + Foundry Local で、この設計のまま工場 Edge に移植可能**。データ非外送 / WAN 断耐性 / 低レイテンシまで届きます。」

**ビジュアル** (画面を縦 3 分割):
- 左 1/3: **3 つの数字** (3 価値 ↔ 3 金額) + 合計 1 つ
- 中 1/3: **チェックマーク 4 つ** (本番デプロイ ✓ / 実コスト数百円 ✓ / 4 段自律性 ✓ / RAI ✓)
- 右 1/3: **簡略図 本社クラウド ⇆ 工場 Edge** (3 制約のラベル)

**話法**:
- 「**3 つの価値ごとに**」 で Slide 2 とリンク
- 「**控えめ**」 を 1 回 (1 回でいい、時間ない)
- 「**この設計のまま移植**」 は強めに ── Foundry を選んだ最大価値

> ⚠️ Foundry Local / Azure Local は **Public Preview** の事実明記、誇張せず。
> 整合: Slide 2 の 3 価値 ↔ Slide 6 の 3 金額。これがピッチ全体を一本の糸にする。

## 5. 想定 Q&A (質疑応答 約 3 分 ── 本番枠)

> **重要**: 3 分の質疑があるので、ピッチで言いきれなかった要素はここで深掘りで取り戻す。
> 特に **「賢くなる(評価ハーネス)」「責任の所在」「Foundry の Edge 移植性」** はピッチで省略しているので、Q&A での扱いを必ず練習する。

### 5-A. 製品・課題への問い

| 突かれそうな点 | 一次回答 |
|---|---|
| **製造現場で働いた経験はあるのか?** | 「ライン作業者経験はありません。ただし **エンジニアとして製造業に近い案件にいくつか入っています**。本プロダクトは Siemens / 経産省 ものづくり白書 などの公開資料の構造的問題を出発点に、案件で観察した症状 (検索コスト・暗黙知・夜間属人化) と突き合わせて作っています」 |
| **オペレーターが直せないのに AI が判断助言する意味は?** | 「**オペレーターは直す担当として想定していません**。AI が縮めるのは **保全の診断時間**。Teams 通知に推奨工具リストと過去事例が同梱されるので、保全は到着直後から動けます。1 出動あたり 15-30 分の短縮」 |
| **現場に持ち込んだら使われないのでは?** | 「**フォーム + 自由記述 + 写真自動入力** で既存の作業ログ習慣に寄せています。チャット UI にしなかったのはそのため。Vision の自動入力でオペレーターの入力負担はゼロに近い」 |
| **保全側がこれを使うインセンティブは?** | 「保全側の **診断時間を短縮** するので、保全こそが受益者。出動回数も減るので、夜間呼び出しコスト削減にも直結する」 |

### 5-B. 設計・自律性への問い

| 突かれそうな点 | 一次回答 |
|---|---|
| **「時間勝負」と言いつつ承認で待つのは矛盾では?** | 「**緊急度で動作を切替**。Critical は承認スキップ・即時。High は 5 秒承認。Medium は承認 + 並行作業。Low は通知せず自己解決。これで両立を構造化 (Slide 5)」 |
| **そんな大事な判断を AI に任せてよいのか?** | 「**全部は任せていません**。Critical 以外は人が必ず承認。Critical の auto-fire も **承認者ログ・送信内容が全件監査** されます」 |
| **Critical の判定基準は LLM 任せでは?** | 「正直、現状は LLM 任せです。プロンプトで定義は明示していますが、明示ルール化(キーワード・センサー閾値ベース)は次の実装課題。**ピッチでは弱みとして開示しておくのが正直**」 |
| **責任の所在は?** | 「**すべての承認に承認者 ID と監査ログ**。Critical の auto-fire でも送信内容・時刻・対象者が全件残ります」 |

### 5-C. 技術・運用への問い

| 突かれそうな点 | 一次回答 |
|---|---|
| **AI が幻覚を出したら?** | 「全主張に出典 (citation) を強制し、**Trust band (red/yellow/green) で出力の groundedness を簡易判定**。red は自動で人に回す (実装済み)」 |
| **使うほど賢くなる、と言うが本当か?** (← ピッチで省略) | 「**評価ハーネスで定量しています**。現場フィードバックの ON/OFF で **Top1 正答率 + groundedness** を比較。N=◯◯ の小規模試験ですが、ON 側が上回ります。フレーム自体は実コーパスに横展開可能 (Foundry Evaluation で継続計測)」 |
| **モデルが古くなる?** | 「現場フィードバック (実原因・復旧時間・AI 的中可否) で自動更新。閉じた静的 RAG ではない」 |
| **Foundry を使う必然性は?** | 「Connected agents の native + 実行トレース observability。Semantic Kernel だと自前で書く可視化が出来合いで揃う。さらに **Foundry Local で Edge にそのまま移植** できるのが決定打」 |
| **Foundry Local / Azure Local は Public Preview だが?** | 「事実そのとおりです。本プロダクトは **クラウドで本番稼働、Edge は将来の移植先として設計を担保**、というスタンス。設計の地続き感が価値です」 |
| **合成コーパスの限界は?** | 「評価フレームは実コーパスに横展開可能。Foundry Evaluation で継続計測する設計」 |
| **コスト試算の楽観性は?** | 「**控えめ側で試算**。前提 3 行を明示。Aberdeen 2.5 万ドル/時、Siemens 1.4 兆ドル/年 を出典に揃えている」 |
| **実コスト数百円というのは本当?** | 「個人クレジット枠内、Agent Service は固定費ゼロ、AOAI のトークン課金のみ。Cosmos Serverless / SWA / App Service B1 で月数百円規模です」 |

### 5-D. 答えにくいやつ (準備しておく)

## 6. スライド 1 枚あたりの設計原則

- **1 スライド = 1 メッセージ**。タイトルは命題形 ("〜である")
- **テキスト 30 字以内 × 3 行まで**。8 分では読まれない
- **画像 > 図 > 表 > テキスト** の優先順
- **色は 3 色まで** (背景 / 強調 / 補助)
- **アイコン > 箇条書き** (受賞作の通則)
- **3 回の法則**: 「**最初の 3 分**」「**1 画面で返る**」「**使うほど賢くなる**」を全編で繰り返す
